小扣在迭代四主要承担流程协调、文档归档和边界复核。重点不是直接实现所有功能，而是把需求、频道、角色、验收证据和最终报告串起来，确保迭代结果可以被复盘和交付。

## 三、做得好的部分

1. 频道和身份先确认再同步。迭代四专用频道明确为 `<#C0BB035G3DK>`，小扣作为 Codex Slack gateway bot 响应，避免了跨 persona 或 DM 路由错误。
2. 文档留在当前项目。ChorusGate 专属报告归档到 `docs/reports`，不外溢到其他项目域。
3. 能力边界及时收敛。Codex 不支持 Claude 等价审批后，文档没有继续包装成“统一审批完成”，而是明确为 sandbox 安全升级。
4. 对测试结论做了二次审视。发现 135/135 只是组件级/集成级通过后，推动补充 6/20 E2E 验收报告。

## 四、暴露的问题

| 问题 | 影响 | 后续处理 |
|------|------|----------|
| 角色、频道、bot 身份容易混淆 | 通知可能发错、报告署名不准 | 项目内继续以 `memory/project-team-channels.md` 为准 |
| 规划文档和实际结论存在滞后 | 例如 Codex 审批从“交互审批”变为“sandbox” | 最终报告中明确“降级完成”，后续更新规划勾选状态 |
| 测试报告分散在 `docs/tests` | 管理层阅读成本高 | 本次新增汇总测试报告到 `docs/reports/v4/` |
| 用户侧功能说明缺口 | 功能完成后使用入口不够集中 | 本次新增用户手册 |

## 五、经验沉淀

- 项目协作类交付，频道 ID、用户 ID、角色名和文档路径都属于质量要求。
- 最终汇报应把“做了什么、验证了什么、没验证什么、下一步是什么”放到同一个读者面。
- 当实现结论推翻早期方案假设时，应在最终报告里正面说明，而不是只改代码。

## 六、下轮建议

2. 把“管理汇报、测试报告、用户手册”列为每个迭代的固定收口清单。
3. 对 Slack 通知继续使用真实 channel ID 和 `<@USERID>`，不使用显示名代替可触达 mention。
4. 后续若需要广播到 `<#C0AHL7U33EE>`，先确认最终报告链接和目标读者，再发送。
